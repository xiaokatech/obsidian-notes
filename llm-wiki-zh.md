# LLM Wiki

使用大语言模型构建个人知识库的模式

这是一个想法文件，设计用于复制粘贴到你自己的 LLM 代理（如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。其目标是传达高层次的想法，但你的代理将与你合作构建具体实现。

## 核心理念

大多数人对 LLM 和文档的体验看起来像 RAG：你上传一组文件，LLM 在查询时检索相关片段，并生成答案。这有效，但 LLM 在每次提问时都是从零开始重新发现知识。没有积累。问一个需要综合五个文档的微妙问题，LLM 每次都必须找到并拼凑相关片段。没有构建任何东西。NotebookLM、ChatGPT 文件上传和大多数 RAG 系统都是这样工作的。

这里的想法不同。LLM 不仅仅在查询时从原始文档中检索，而是增量地构建和维护一个持久的 wiki——一个结构化的、相互链接的 markdown 文件集合，位于你和原始来源之间。当你添加一个新来源时，LLM 不只是为了后续检索而索引它。它读取它，提取关键信息，并将其集成到现有的 wiki 中——更新实体页面，修订主题摘要，指出新数据与旧主张矛盾的地方，加强或挑战不断发展的综合。知识被编译一次，然后保持最新，而不是在每次查询时重新推导。

这是关键区别：wiki 是一个持久的、复合的产物。交叉引用已经存在。矛盾已经被标记。综合已经反映了你阅读的所有内容。wiki 随着你添加的每个来源和提出的每个问题而变得更加丰富。

你从不（或很少）自己编写 wiki——LLM 编写和维护所有内容。你负责来源、探索和提出正确的问题。LLM 完成所有繁琐的工作——总结、交叉引用、归档和簿记，这些工作使知识库随着时间的推移真正有用。在实践中，我一边打开 LLM 代理，另一边打开 Obsidian。LLM 根据我们的对话进行编辑，我实时浏览结果——跟踪链接、检查图形视图、阅读更新的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这可以应用于许多不同的场景。一些例子：

- **个人**：追踪自己的目标、健康、心理、自我提升——归档日记条目、文章、播客笔记，并随着时间的推移构建一个结构化的自我图景。
- **研究**：深入一个主题数周或数月——阅读论文、文章、报告，并增量地构建一个具有不断发展的论文的全面 wiki。
- **阅读一本书**：在阅读过程中归档每一章，为角色、主题、情节线以及它们如何连接构建页面。最终你会得到一个丰富的伴侣 wiki。想想像 Tolkien Gateway 这样的粉丝 wiki——数千个相互链接的页面涵盖角色、地点、事件、语言，由志愿者社区多年构建。你可以在阅读时个人构建类似的东西，LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，由 Slack 线程、会议记录、项目文档、客户电话提供信息。可能有人类在循环中审查更新。wiki 保持最新，因为 LLM 完成了团队中没有人想做的维护工作。
- **竞争分析、尽职调查、旅行规划、课程笔记、爱好深入研究**——任何你在积累知识并希望它被组织而不是分散的场景。

## 架构

有三个层次：

1. **原始来源**——你策划的源文档集合。文章、论文、图像、数据文件。这些是不可变的——LLM 从中读取但从不修改它们。这是你的真理来源。

2. **Wiki**——一个 LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、比较、概述、综合。LLM 完全拥有这一层。它创建页面，在新来源到达时更新它们，维护交叉引用，并保持一切一致。你阅读它；LLM 编写它。

3. **模式**——一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM wiki 的结构如何，约定是什么，以及在摄取来源、回答问题或维护 wiki 时要遵循的工作流程。这是关键的配置文件——它使 LLM 成为一个有纪律的 wiki 维护者，而不是一个通用的聊天机器人。你和 LLM 随着时间的推移共同演化这个文件，以找出适合你领域的方法。

## 操作

### 摄取

你将一个新来源放入原始集合中，并告诉 LLM 处理它。一个示例流程：LLM 读取来源，与你讨论关键要点，在 wiki 中编写摘要页面，更新索引，更新整个 wiki 中的相关实体和概念页面，并将条目附加到日志中。单个来源可能涉及 10-15 个 wiki 页面。我个人更喜欢一次摄取一个来源并保持参与——我阅读摘要，检查更新，并指导 LLM 强调什么。但你也可以批量摄取许多来源，监督较少。这取决于你开发适合你风格的工作流程，并在模式中记录以供未来会话使用。

### 查询

你针对 wiki 提问。LLM 搜索相关页面，读取它们，并综合一个带有引用的答案。答案可以根据问题的不同采取不同的形式——markdown 页面、比较表格、幻灯片（Marp）、图表（matplotlib）、画布。重要的见解：好的答案可以作为新页面归档回 wiki。你要求的比较、分析、你发现的连接——这些是有价值的，不应该消失在聊天历史中。这样，你的探索就像摄取的来源一样在知识库中复合。

### 检查

定期要求 LLM 对 wiki 进行健康检查。寻找：页面之间的矛盾、新来源已取代的过时主张、没有入站链接的孤立页面、提到但缺少自己页面的重要概念、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 擅长建议调查的新问题和寻找的新来源。这使 wiki 在增长过程中保持健康。

## 索引和日志记录

两个特殊文件帮助 LLM（和你）在 wiki 增长时导航。它们有不同的用途：

- **index.md**是内容导向的。它是 wiki 中所有内容的目录——每个页面都列有链接、一行摘要，以及可选的元数据，如日期或来源计数。按类别组织（实体、概念、来源等）。LLM 在每次摄取时更新它。在回答查询时，LLM 首先读取索引以找到相关页面，然后深入其中。这在中等规模（约 100 个来源，约数百个页面）下效果出奇地好，并避免了基于嵌入的 RAG 基础设施的需求。

- **log.md**是按时间顺序的。它是发生了什么以及何时发生的仅追加记录——摄取、查询、检查过程。一个有用的提示：如果每个条目以一致的前缀开头（例如`## [2026-04-02] ingest | Article Title`），日志就可以用简单的 unix 工具解析——`grep "^## \[" log.md | tail -5`给你最后 5 个条目。日志给你 wiki 演化的时间线，并帮助 LLM 理解最近做了什么。

## 可选：CLI 工具

在某些时候，你可能想要构建小工具来帮助 LLM 更有效地操作 wiki。最明显的是 wiki 页面的搜索引擎——在小规模下，索引文件就足够了，但随着 wiki 的增长，你需要适当的搜索。qmd 是一个不错的选择：它是一个用于 markdown 文件的本地搜索引擎，具有混合 BM25/向量搜索和 LLM 重新排名，全部在设备上。它既有 CLI（因此 LLM 可以调用它），也有 MCP 服务器（因此 LLM 可以将其用作本地工具）。你也可以自己构建更简单的东西——LLM 可以根据需要帮助你编写一个简单的搜索脚本。

## 技巧和窍门

- **Obsidian Web Clipper**是一个浏览器扩展，将网页文章转换为 markdown。对于快速将来源放入原始集合非常有用。
- **本地下载图像**。在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设置为固定目录（例如 raw/assets/）。然后在设置 → 热键中，搜索"Download"找到"Download attachments for current file"并将其绑定到热键（例如 Ctrl+Shift+D）。剪辑文章后，按热键，所有图像都会下载到本地磁盘。这是可选的但很有用——它让 LLM 直接查看和引用图像，而不是依赖可能失效的 URL。注意，LLM 不能原生地一次性读取带有内联图像的 markdown——解决方法是让 LLM 先读取文本，然后单独查看部分或全部引用的图像以获得额外上下文。这有点笨拙，但足够有效。
- **Obsidian 的图形视图**是查看 wiki 形状的最佳方式——什么连接到什么，哪些页面是中心，哪些是孤立的。
- **Marp**是一种基于 markdown 的幻灯片格式。Obsidian 有一个插件。对于直接从 wiki 内容生成演示文稿很有用。
- **Dataview**是一个 Obsidian 插件，可以查询页面 frontmatter。如果你的 LLM 向 wiki 页面添加 YAML frontmatter（标签、日期、来源计数），Dataview 可以生成动态表格和列表。
- wiki 只是一个 markdown 文件的 git 仓库。你可以免费获得版本历史、分支和协作。

## 为什么这有效

维护知识库的繁琐部分不是阅读或思考——而是簿记。更新交叉引用，保持摘要最新，指出新数据与旧主张矛盾的地方，保持跨数十个页面的一致性。人类放弃 wiki 是因为维护负担的增长速度超过了价值。LLM 不会感到无聊，不会忘记更新交叉引用，并且可以在一次处理中触及 15 个文件。wiki 保持维护，因为维护成本几乎为零。

人类的工作是策划来源、指导分析、提出好问题，并思考这一切意味着什么。LLM 的工作是其他一切。

这个想法在精神上与 Vannevar Bush 的 Memex（1945）相关——一个个人的、策划的知识存储，文档之间有联想轨迹。Bush 的愿景更接近这个，而不是网络变成的样子：私人的、积极策划的，文档之间的连接与文档本身一样有价值。他无法解决的部分是谁来做维护。LLM 处理了这个问题。

## 注意

本文档故意抽象。它描述了想法，而不是具体实现。确切的目录结构、模式约定、页面格式、工具——所有这些都取决于你的领域、你的偏好和你选择的 LLM。上面提到的所有内容都是可选和模块化的——选择有用的，忽略无用的。例如：你的来源可能只是文本，所以你根本不需要图像处理。你的 wiki 可能足够小，索引文件就是你需要的全部，不需要搜索引擎。你可能不关心幻灯片，只想要 markdown 页面。你可能想要一组完全不同的输出格式。正确使用这个的方法是将其与你的 LLM 代理分享，并合作实例化一个适合你需求的版本。本文档的唯一工作是传达模式。你的 LLM 可以解决其余问题。
